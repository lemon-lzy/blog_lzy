# java中常见的集合类？
### 集合类分为两大类：
Collection 接口和 Map 接口，Collection 存单个元素，Map 存键值对。
### Map:
Map 接口常用的实现类有 HashMap，线程安全的 ConcurrentHashMap。
### Collection:
而 Collection 接口又分为三个子接口：List、Set 和 Queue。

- List 是有序可重复集合，常见的实现类有：动态数组实现的ArrayList、双向链表实现的LinkedList、线程安全的CopyOnWriteList。

- Set 是无重复集合，常见实现类有：HashSet、记录插入顺序的 LinkedHashSet、有序的 TreeSet。

- Queue 是先入先出队列，常见实现类有：优先级队列 PriorityQueue，优先级高的先出，LinkedList 也实现了 Queue 接口，可以当普通队列使用。


# 集合的使用场景与实现
- 存取有序用List
- 希望自动去掉重复元素用Set
- 通过唯一的key找value用map
- 队列先进先出用Queue
- 栈后进先出用Stack


# Collections和Collection的区别
### Collection
- Collection是Java集合框架中的一个接口，它是所有单对象集合类的基础接口。它定义了一组通用的操作和方法，如添加、删除、遍历等，用于操作和管理一组对象。
- Collection接口有许多实现类，如List、Set和Queue等。
### Collections
Collections是Java提供的一个**工具类**，位于**java.util**包中
- 它提供了一系列**静态方法**，用于对集合进行操作和算法,包括 排序、查找、替换、反转、随机化等等。
- 这些方法可以对**实现了Collection接口的集合**进行操作，如List和Set类似于各种工具包中的CollectionUtils.

# ArrayList的扩容机制？
- 当new一个ArrayList的时候，此时容量为0。当第一次插入元素的时候，扩容为10。
- 如果元素大小超过数组大小时就会触发扩容，创建一个大小为原来1.5倍的新数组，然后通过**Arrays.copyOf**的方法将元素拷贝新数组去

# ArrayList和LinkedList的区别？
ArrayList是通过**动态数组**实现的，**内存空间连续**。

- 随机访问时可以通过基地址+下标*元素大小获取，所以随机访问速度快。

- 插入删除需要移动后续节点，而且又有可能触发扩容，所以插入删除比较慢。

LinkList是基于**双向链表**进行实现的，**内存空间不连续**。

- 随机访问需要遍历链表，所以随机访问比较慢，
- 而插入删除只需要改变指针指向，所以插入删除会比较快。

### 但是java的LinkedList的插入删除的性能其实很多场景比ArrayList要差。
- 原因之一因为Java的LinkedList在插入删除时其实找节点也需要耗时。
- 还有就是因为cpu的预读机制，在读取一个元素时会预读一块内存然后缓存到cpu三级缓存，所以ArrayList因为内存连续，在移动元素操作时很容易命中三级缓存。

# HashMap和Hashtabe的区别
- hashMap线程不安全，Hashtbale使用全局锁保证线程安全。
- HashMap允许null键null值，hashtable不允许null键null值。

# CopyOnWriteList?
CopyOnWriteArrayList 是应该线程安全 List，采用写时复制机制保证线程安全。
### 核心原理
- 读操作：不加锁，直接读，并发读性能极高。
- 写操作（add/set/remove）：
复制一份新数组;
在新数组上执行修改;
再将原数组引用指向新数组;
整个写过程加锁，保证同一时间只有一个线程在写，不会出现并发修改异常。

# 什么是哈希冲突，怎么解决？
### 哈希冲突
指的是不同的对象计算出来的哈希值相同，则发生了哈希冲突
### 解决办法
- **拉链法**:将冲突的元素存在同一个链表
- **开发寻址法**:寻找下一个可用位置
- **再哈希法**:用另一个哈希函数重新计算

# 说下HashMap的原理/地层结构？

HashMap是java的一个键值对集合，支持一个null键和多个null值。底层在**JDK7**是使用**数组加链表**，**JDK8**及以上是使用**数组+链表+红黑树**组成，初始容量是0，当第一个元素插入的时候会扩容为16。

当插入一个元素的时候，会计算Key的hash值然后和数组长度进行&运算得到对应的数组下标，当发生哈希冲突的时候，使用链表进行解决，JDK7是头插法，JDK8是尾插法。

当链表长度≥8且数组长度≥64的时候，会转化为红黑树，在≤6时候会退化为链表。

在插入后会检查当前元素是否大于数组容量*负载因子（默认0.75），如果是的话，则会触发扩容，创建一个长度为旧数组两倍的数组，然后将旧数组元素搬到新数组。这里在JDK7和JDK8也不一样，JDK7是重新计算哈希值，确定在新数组的位置；JDK8是通过将哈希值和旧数组长度进行&运算，然后看第一位，如果是0，则新数组位置就是原位置，1则新数组位置是原位置+旧数组长度。

## 为什么会用红黑树？
- 当哈希冲突较多时，链表中的元素增多，查找、插入和删除的时间复杂度从 O(1) 退化为 O(n)。
- 而如果用红黑树会将查询，插入和删除的复杂度降低到O(logN)。

## HashMap为什么要使用红黑树, 为啥不用链表/平衡二叉树
### 为什么不用链表:
- 当某个位置, Hash冲突严重, 则链表的长度会很长. 那么查找的时候依次比较, 效率会很低 O(n).
- 将链表转为红黑树, 因为红黑树是排序树, 查找效率一般是O(logN), 使用红黑树就提高了查找效率.
### 为什么不用平衡二叉树:
- 平衡二叉树追求绝对平衡，自平衡需要旋转的次数可能很大
- 红黑树追求大致平衡，自平衡需要的旋转次数不超过三次

## 为何一上来不树化
- 刚开始链表的长度可能只有三四个, 如果此时树化, 那么树化后查询的效率和短链表差不多, 意义不大.
- 而且链表的节点是Node, 红黑树的节点为TreeNode. TreeNode的内存占用大于Node. 所以非必要不树化

## 为何树化阈值为8
- 如果阈值太小的话，可能链表到红黑树的查询效率提升不明显
- 链表的节点是Node, 红黑树的节点为TreeNode. TreeNode的内存占用大于Node. 所以非必要不树化
- 长度超过8的链表出现几率非常小（千万分之六）, 选择8就是为了让树化几率足够小

## 为什么定了 6 而不是小于等于 8 就变？
是因为要留个缓冲余地，避免反复在树与链表之间转化。

## hashCode有了, 为何还有hash()方法?
二次hash是为了让hash值分布更加均匀, 减少hash冲突, 从而
使链表更短, 因此也就提升了查找效率.
## 数组容量为何是2^n?
- 计算索引时，如果是2的n次方可以使用位与运算代替取模运算, 效率更高
- HashMap 扩容时，通过容量为 2 的 n 次方，扩容时只需通过原哈希值与原数组长度判断是否需要迁移，这减少了重新计算索引的开销，提升了 rehash 的效率。

## 负载因子为何是0.75
0.75是在占用空间和查询时间中取得了比较好的平衡. (扩容阈值=数组大小*负载因子)
- 大于0.75, 冲突增加了, 数组空间就节省了, 但是链表就会比较长,影响性能.
- 小于0.75, 冲突减少了, 链表会比较短, 数组扩容会很频繁, 空间占用增多.

## 多线程操作HashMap会出现什么问题
知道hashMap是线程不安全即可. 具体并发性会出现什么问题了解即可.

## hashMap在1.7到1.8除了红黑树还有什么改动？
怎么记？在计算哈希值、插入和插入后扩容三个阶段的优化：

1. **优化哈希计算**:由原来对 key 的 hashcode 进行四次异或运算+移位改为对 key 的 hashcode 的高十六位和低十六位进行异或运算，使哈希分布更加均匀。

2. **将头插法改为尾插法**:减少并发情况下成环的概率。

3. **优化扩容机制**:原来扩容时元素的 hash 值都要重新计算，优化后不需要重新计算，而是用 hash & 老数组长度，判断最高位是 0 还是 1，0 的话就留在老数组的下标，1 则改为老数组位置+老数组的下标。

# ConcurrentHashMap
## ConcurrentHashMap与HashMap的区别是什么？
ConcurrentHashMap是HashMap的升级版，HashMap是线程不安全的而ConcurrentHashMap是线程安全。而其他功能和实现原理和HashMap类似

## ConcurrentHashMap与Hashtable的区别是什么？
### Hashtable:
Hashtable也是线程安全的，整个Hashtable对应一把锁，同
时刻，只能有一个线程操作它, 并发性低
### ConcurrentHashMap
- 1.7的ConcurrentHashMap是使用分段锁（继承ReenTranLock）解决线程安全问题，将map分为多个段对每个段进行加锁,默认分为16段。
- 而1.8是通过CAS加synchronized解决线程安全问题，当插入数据的时候，如果该索引下面没有元素，则使用CAS进行插入，如果有，则在该位置的头节点进行加锁然后插入，锁的粒度更细，比起1.7并发性能更好

## 1.7和1.8之间的区别？
### 数据结构不同:
- 1.7的ConCurrentaHashMap用的是数组加链表的结构
- 而1.8是数组加链表加红黑树。
### 锁的粒度不同:
- 1.7的ConcurrentHashMap是使用分段锁（继承ReenTranLock）解决线程安全问题，将map分为多个段对每个段进行加锁,默认分为16段。
- 而1.8是通过CAS加synchronized解决线程安全问题，当插入数据的时候，如果该索引下面没有元素，则使用CAS进行插入，如果有，则在该位置的头节点进行加锁然后插入，锁的粒度更细，比起1.7并发性能更好

## ConcurrentHashMap 的 get 方法是否要加锁，为什么？
get 方法不需要加锁。
- 因为 Node 的元素 value 和指针 next 是用 volatile 修饰的
- 在多线程环境下线程A修改节点的 value 或者新增节点的时候是对线程B可见的。

## ConcurrentHashMap 的并发度是什么
- 在JDK1.7中，实际上就是ConcurrentHashMap中的分段锁个数，即Segment[]的数组长度，默认是16，这个值可以在构造函=数中设置。
- 在JDK1.8中，已经摒弃了Segment的概念，选择了Node数组+链表+红黑树结构，并发度大小依赖于数组的大小
