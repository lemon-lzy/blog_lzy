# 设计模式分类？
23 种设计模式按照解决问题的类型分成三大类：**创建型**、**结构型**、**行为型**。

1）创建型模式解决的是"对象怎么创建"的问题。核心目的是把对象的创建过程和使用过程分开，避免代码里到处都是 new 和复杂的初始化逻辑。包括单例模式、工厂方法模式、抽象工厂模式、建造者模式、原型模式这 5 种。其中单例、工厂、建造者在实际项目里用得最多。

2）结构型模式解决的是"对象怎么组合"的问题。核心目的是通过组合不同的类或对象，形成更大、更灵活的结构。包括适配器模式、桥接模式、组合模式、装饰器模式、外观模式、享元模式、代理模式这 7 种。适配器、装饰器、代理在日常开发中最常见。

3）行为型模式解决的是"对象怎么协作"的问题。核心目的是定义对象之间的通信方式和职责分配。包括责任链模式、命令模式、迭代器模式、中介者模式、备忘录模式、观察者模式、状态模式、策略模式、模板方法模式、访问者模式、解释器模式这 11 种。策略、模板方法、观察者、责任链用得比较多。


# 单例模式
单例模式是一种创建型设计模式，核心目的是保证一个类在整个程序运行期间只有一个实例。

**单例设计模式分为两种:懒汉式和硬汉式**
### 懒汉式：
类加载时实例对象就被创建
### 硬汉式:
类实例真正被使用时才被创建
## 实现
单例模式的实现有两个关键点：

1）构造函数私有化。不让外部直接 new，断掉从外面创建实例的路。

2）类内部自己持有唯一实例。提供一个静态方法让外部获取这个实例，每次调用都返回同一个对象。
### 懒汉式-静态变量实现;
````
public class Singleton {

    // 1，私有构造方法
    private Singleton() {}

    // 2，在本类中创建本类对象
    private static Singleton instance = new Singleton();

    // 3，提供一个公共的访问方式，让外界获取该对象
    public static Singleton getInstance() {
        return instance;
    }
}
````   
### 静态内部类;
````
public class Singleton {
    private Singleton() {}

    private static class Holder {
        private static final Singleton INSTANCE = new Singleton();
    }

    public static Singleton getInstance() {
        return Holder.INSTANCE;
    }
}

````
### 枚举：
````
public enum Singleton {
    INSTANCE;

    public void doSomething() {
        // 业务逻辑
    }
}
````

### 双重检查锁
````
public class Singleton {
    private static volatile Singleton instance; // 使用 volatile 防止指令重排

    private Singleton() {}

    public static Singleton getInstance() {
        if (instance == null) { // 第一次检查：避免不必要的同步
            synchronized (Singleton.class) {
                if (instance == null) { // 第二次检查：确保实例唯一
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
````

### 为什么双重检查锁定需要 volatile 关键字？
在 Java 中，volatile 修饰符用于防止指令重排序，从而确保双重检查锁定的正确性。

instance = new Singleton() 是一个非原子操作，它分为以下三步：

- 分配内存空间。
- 初始化对象。
- 将对象的引用赋值给 instance。
在没有 volatile 的情况下，编译器和 CPU 可能会对这些步骤进行重排序（比如执行顺序变成 1 → 3 → 2）。此时，另一个线程可能会在 instance 被赋值后，但对象尚未完成初始化时访问它，从而导致错误。

所以将 instance 声明为 volatile，可以禁止指令重排序，确保对象的初始化过程对所有线程可见。

# 什么是策略模式？
策略模式是一种行为型的设计模式，策略模式就是定义了一系列的算法，将算法封装起来，然后算法之间可以相互替换，我们想要什么算法就new什么算法就行，不需要写大量id else判断。
## 代码结构：
````
// 1.策略接口
public interface PayStrategy {
    void pay();
}

// 2.具体策略
public class AliPay implements PayStrategy {
    public void pay() { 支付宝支付 }
}
public class WxPay implements PayStrategy {
    public void pay() { 微信支付 }
}

// 3.上下文
public class PayContext {
    private PayStrategy strategy;
    public PayContext(PayStrategy strategy) {
        this.strategy = strategy;
    }
    public void executePay() {
        strategy.pay();
    }
}

// 使用
new PayContext(new AliPay()).executePay();
new PayContext(new WxPay()).executePay();
````

# 工厂模式
## 定义：
简单工厂模式的核心思想是把对象的创建逻辑集中到一个工厂类里，调用方只需要告诉工厂"我要什么类型"，工厂根据参数决定创建哪个具体对象返回。
比如我在EasyCode这个项目中就用到了工厂模式根据不同的需求去创建不同的AIService实例。
### 简单工厂：
````
// 产品接口
public interface Database {
    void connect();
}

// 具体产品
public class MySQL implements Database {
    public void connect() {
        System.out.println("连接 MySQL 数据库");
    }
}

public class PostgreSQL implements Database {
    public void connect() {
        System.out.println("连接 PostgreSQL 数据库");
    }
}

// 简单工厂
public class DatabaseFactory {
    public static Database createDatabase(String type) {
        switch (type) {
            case "mysql":
                return new MySQL();
            case "postgresql":
                return new PostgreSQL();
            default:
                throw new IllegalArgumentException("不支持的数据库类型: " + type);
        }
    }
}

// 客户端使用
Database db = DatabaseFactory.createDatabase("mysql");
db.connect();
````
### 为什么要用简单工厂
假设没有工厂，代码里到处都是这样的创建逻辑：
````
Database db;
if (config.getType().equals("mysql")) {
db = new MySQL();
} else if (config.getType().equals("postgresql")) {
db = new PostgreSQL();
}
````
这段代码可能出现在 10 个不同的类里。有一天要加个 Oracle 支持，你得改 10 个地方,创建对象的过程和业务逻辑耦合，用简单工厂把创建逻辑收拢到一处，以后加新类型只改工厂类一个地方就够了。

### 工厂方法
````
// 1. 工厂接口
public interface DatabaseFactory {
    Database createDatabase();
}

// 2. MySQL工厂
public class MySQLFactory implements DatabaseFactory {
    @Override
    public Database createDatabase() {
        return new MySQL();
    }
}

// 3. PostgreSQL工厂
public class PostgreSQLFactory implements DatabaseFactory {
    @Override
    public Database createDatabase() {
        return new PostgreSQL();
    }
}

// 客户端使用
DatabaseFactory factory = new MySQLFactory();
Database db = factory.createDatabase();
db.connect();
````
### 为什么使用工厂方法：
使用工厂方法，如果需要加一个创建只需要新加一个工厂类，不用去修改，符合开闭原则
### 抽象工厂实现：
````
// 新增产品接口
public interface Transaction {
    void commit();
}

public class MySQLTransaction implements Transaction {
    @Override
    public void commit() {
        System.out.println("MySQL 事务提交");
    }
}

public class PostgreSQLTransaction implements Transaction {
    @Override
    public void commit() {
        System.out.println("PostgreSQL 事务提交");
    }
}

// ==================== 抽象工厂 ====================
public interface AbstractDatabaseFactory {
    Database createDatabase();
    Transaction createTransaction();
}

// MySQL 整套工厂
public class MySQLFactory implements AbstractDatabaseFactory {
    @Override
    public Database createDatabase() {
        return new MySQL();
    }

    @Override
    public Transaction createTransaction() {
        return new MySQLTransaction();
    }
}

// PostgreSQL 整套工厂
public class PostgreSQLFactory implements AbstractDatabaseFactory {
    @Override
    public Database createDatabase() {
        return new PostgreSQL();
    }

    @Override
    public Transaction createTransaction() {
        return new PostgreSQLTransaction();
    }
}

// 客户端使用
AbstractDatabaseFactory factory = new MySQLFactory();
Database db = factory.createDatabase();
Transaction tx = factory.createTransaction();

db.connect();
tx.commit();
````
### 抽象工厂和工厂方法区别？
抽象工厂一个工厂类能生产多个产品。工厂方法只能生成一个产品。
# 模版方法模式
模板方法模式是一种行为型设计模式，核心思想是在抽象类中定义算法骨架,定死执行顺序和通用逻辑，把可变的步骤延迟到子类实现。比如我在OJ项目中实现代码沙箱功能就使用到了模版方法模式。


### 整个模式由两个角色组成：

1）**AbstractClass 是抽象类**，里面有个 final 修饰的模板方法定义算法骨架，调用一系列步骤方法。其中有些步骤是抽象方法强制子类实现，有些是默认实现子类可选覆盖

2）**ConcreteClass 是具体子类**，实现抽象方法，按需覆盖钩子方法
### 具体实现
````
// 抽象类
abstract class DataProcessor {
    // 模板方法，定死执行顺序
    public final void process() {
        readData();
        processData();
        writeData();
    }

    protected abstract void readData();    // 子类必须实现
    protected abstract void processData(); // 子类必须实现

    protected void writeData() {           // 默认实现，子类可覆盖
        System.out.println("Writing data to output.");
    }
}

// CSV 处理器
class CSVDataProcessor extends DataProcessor {
    @Override
    protected void readData() {
        System.out.println("Reading data from CSV file.");
    }

    @Override
    protected void processData() {
        System.out.println("Processing CSV data.");
    }
}

// JSON 处理器
class JSONDataProcessor extends DataProcessor {
    @Override
    protected void readData() {
        System.out.println("Reading data from JSON file.");
    }

    @Override
    protected void processData() {
        System.out.println("Processing JSON data.");
    }
}
````



# 代理模式
代理模式是一种结构型设计模式，核心思想是在不改变原始对象的前提下，通过一个代理对象来访问原始对象，对原始对象进行增强，比如权限校验、日志记录、缓存等，其实AOP使用了这种设计模式。

# 观察者模式：
观察者模式是一种行为型设计模式，核心思想是一对多的依赖关系：当一个对象状态发生变化时，所有依赖它的对象都会收到通知并自动更新。

### 典型的应用场景：

1）事件驱动系统，比如 GUI 中按钮点击事件触发多个监听器响应

2）消息推送，像股票 App 价格变动时推送给几十万订阅用户

3）MVC 架构中 Model 层数据变化自动同步到多个 View

### 例子：
在使用微信公众号时，大家都会有这样的体验，当你关注的公众号中有新内容更新的话，它就会推送给关注公众号的微信用户端。我们使用观察者模式来模拟这样的场景，微信用户就是观察者，微信公众号是被观察者，有多个的微信用户关注了程序猿这个公众号。

**1. 抽象主题（Abstract Subject）**
   - **角色**：公众号（抽象概念）
   - **作用**：定义了关注、取消关注、推送消息的接口。
   - **对应代码逻辑**：包含 attach()（关注）、detach()（取消关注）、notify()（通知粉丝）方法的接口或抽象类。
**2. 具体主题（Concrete Subject）**
   - **角色**：程序猿公众号（具体的公众号实例）
   - **作用**：实现抽象主题的接口，内部维护一个粉丝列表，当有新文章发布时，遍历列表通知所有粉丝。
   - **对应代码逻辑**：具体的 YanWuZhangOfficialAccount 类，负责存储粉丝列表，并且在 update() 被调用时执行具体的推送逻辑。
**3. 抽象观察者（Abstract Observer）**
   - **角色**：微信用户（抽象概念）
   - **作用**：定义了接收消息的更新接口。
   - **对应代码逻辑**：包含 update(String message) 方法的接口，所有粉丝都必须实现这个接口才能接收推送。
**4. 具体观察者（Concrete Observer）**
   - **角色**：具体的微信用户（每一个关注的人）
   - **作用**：实现抽象观察者的接口，存储在具体主题的列表中，当接收到推送通知时，执行具体的查看 / 接收动作。
   - **对应代码逻辑**：具体的 User 类，实现了 update() 方法，比如打印出 “我收到了：[文章标题]”。

   **一句话对应关系：**
   - 抽象观察者 = 微信用户接口
   - 具体观察者 = 具体的微信用户（张三、李四）
   - 抽象主题 = 公众号接口
   - 具体主题 = 程序猿公众号

### 实现
````
// 观察者接口
interface Observer {
    void update(String message);
}

// 被观察者接口
interface Subject {
    void addObserver(Observer observer);
    void removeObserver(Observer observer);
    void notifyObservers();
}

// 具体被观察者
class ConcreteSubject implements Subject {
    private List<Observer> observers = new ArrayList<>();
    private String state;

    @Override
    public void addObserver(Observer observer) {
        observers.add(observer);
    }

    @Override
    public void removeObserver(Observer observer) {
        observers.remove(observer);
    }

    @Override
    public void notifyObservers() {
        for (Observer observer : observers) {
            observer.update(state);
        }
    }

    public void setState(String state) {
        this.state = state;
        notifyObservers();
    }
}

// 具体观察者
class ConcreteObserver implements Observer {
    private String name;

    public ConcreteObserver(String name) {
        this.name = name;
    }

    @Override
    public void update(String message) {
        System.out.println(name + " received update: " + message);
    }
}
````
其实就是被观察者（主题）维护了一个观察者列表，然后当被观察者（主图）发生变化，就循环调用维护的观察者列表的更新逻辑

### 好处：
实现了观察者和被观察者的一个解耦合

### 观察者模式中，如果某个观察者的 update 方法抛异常了，会影响其他观察者收到通知吗？
**回答：**默认实现会影响。因为 notifyObservers 通常是遍历列表依次调用，一个抛异常后面的就行不执了。解决办法是在遍历里加 try-catch，每个观察者单独捕获异常，保证一个出问题不影响其他的。Spring 的 ApplicationEventMulticaster 就是这么处理的。

### 在分布式系统中，观察者模式还适用吗？
**回答：**直接用不太合适，因为观察者模式是进程内的同步调用，跨进程跨网络没法直接调用。分布式场景一般用消息队列来替代，本质上是发布订阅模式。比如订单服务创建订单后往 RocketMQ 发消息，库存服务、积分服务各自订阅这个 Topic 来处理，服务之间完全解耦。

# 建造者模式：
建造者模式是一种创建型设计模式，核心目的是把复杂对象的创建过程拆开，一步步组装，代码更清晰.

### 建造者模式主要包含四个角色：

1）Product 产品类，就是我们要创建的复杂对象，可能包含多个组成部分

2）Builder 抽象建造者，定义了构建产品各个部分的抽象方法

3）ConcreteBuilder 具体建造者，实现 Builder 接口，负责具体怎么构建每个部件

建造者模式是一种创建型设计模式，核心目的是把复杂对象的构建过程和最终表示分离开。通过一步一步地构建对象，既能让构建过程清晰可控，又能用同一套构建流程造出不同的对象表示。

### 建造者模式主要包含四个角色：

1）**Product 产品类**，就是我们要创建的复杂对象，可能包含多个组成部分

2）**Builder 抽象建造者**，定义了构建产品各个部分的抽象方法

3）**ConcreteBuilder 具体建造者**，实现 Builder 接口，负责具体怎么构建每个部件

4）**Director 指挥者**，负责调用建造者的方法，按照特定顺序组装产品

### 建造者模式解决了什么痛点
传统创建对象的方式主要有两种：构造器和 setter。但它们各有硬伤。

构造器方式，参数一多就抓瞎：
````
User user = new User("张三", 25, true, null, "北京", null, false, 1, 2, 3, "工程师", false, null, null, true);
````
15 个参数排一排，谁能记住第 8 个是什么意思？传错了还不好排查。

#### setter 方式看着清晰，但有三个问题：

1）没法做必填校验，哪些字段必须填、哪些可以不填，调用方压根不知道

2）没法做属性间的关联校验，比如开始时间必须小于结束时间

3）对象创建出来是可变的，后面随时可能被改，在并发场景下不安全

建造者模式一次性解决这些问题：链式调用可读性好，build 方法里可以做各种校验，最后返回的是不可变对象。

### 代码实现：
````
// 产品类
class Product {
    private String partA;
    private String partB;

    public void setPartA(String partA) {
        this.partA = partA;
    }

    public void setPartB(String partB) {
        this.partB = partB;
    }

    @Override
    public String toString() {
        return "Product [PartA=" + partA + ", PartB=" + partB + "]";
    }
}

// 抽象建造者
abstract class Builder {
    protected Product product = new Product();

    public abstract void buildPartA();
    public abstract void buildPartB();

    public Product getResult() {
        return product;
    }
}

// 具体建造者
class ConcreteBuilder extends Builder {
    @Override
    public void buildPartA() {
        product.setPartA("Custom Part A");
    }

    @Override
    public void buildPartB() {
        product.setPartB("Custom Part B");
    }
}

// 指挥者
class Director {
    private Builder builder;

    public Director(Builder builder) {
        this.builder = builder;
    }

    public void construct() {
        builder.buildPartA();
        builder.buildPartB();
    }
}

// 客户端
public class Main {
    public static void main(String[] args) {
        Builder builder = new ConcreteBuilder();
        Director director = new Director(builder);
        director.construct();

        Product product = builder.getResult();
        System.out.println(product);
    }
}
````
# 责任链模式:
### 定义：

责任链模式是一种行为型设计模式，它将多个处理器对象串联成一条链，请求从链的起点开始传递，每个处理器要么处理该请求，要么将其传递给链中的下一个处理器，直到请求被处理完毕。

### 核心好处：

- 解耦：请求发送方无需关心具体由哪个处理器处理，只需将请求提交到链中。

- 灵活：可以动态调整链中节点的顺序，或增删处理节点，扩展性强。

### 实现：
````
// 请假请求
class LeaveRequest {
    private String name;
    private int days;

    public LeaveRequest(String name, int days) {
        this.name = name;
        this.days = days;
    }

    public String getName() { return name; }
    public int getDays() { return days; }
}

// 抽象审批人
abstract class Approver {
    protected Approver next;

    public void setNext(Approver next) {
        this.next = next;
    }

    public abstract void handleRequest(LeaveRequest request);
}

// 组长
class Leader extends Approver {
    @Override
    public void handleRequest(LeaveRequest request) {
        if (request.getDays() <= 3) {
            System.out.println("组长审批通过：" + request.getName());
        } else if (next != null) {
            next.handleRequest(request);
        }
    }
}

// 经理
class Manager extends Approver {
    @Override
    public void handleRequest(LeaveRequest request) {
        if (request.getDays() <= 7) {
            System.out.println("经理审批通过：" + request.getName());
        } else if (next != null) {
            next.handleRequest(request);
        }
    }
}

// 老板
class Boss extends Approver {
    @Override
    public void handleRequest(LeaveRequest request) {
        System.out.println("老板最终审批：" + request.getName());
    }
}

// 测试
public class Test {
    public static void main(String[] args) {
        Approver leader = new Leader();
        Approver manager = new Manager();
        Approver boss = new Boss();

        leader.setNext(manager);
        manager.setNext(boss);

        leader.handleRequest(new LeaveRequest("张三", 2));
        leader.handleRequest(new LeaveRequest("李四", 5));
        leader.handleRequest(new LeaveRequest("王五", 10));
    }
}
````
