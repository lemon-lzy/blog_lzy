# Http1.0、1.1、2.0、3.0之间的区别？

# Http请求包括了哪些内容？
### 一、HTTP 请求结构
HTTP 请求由 请求行、请求头、空行、请求体 四部分组成：
### 请求行
包含：请求方法、请求路径、HTTP 协议版本
### 请求头
以 Key: Value 形式描述客户端信息、请求参数等
### 空行
固定分隔符，用于区分请求头与请求体
### 请求体
客户端提交给服务端的数据，如表单、JSON、文件等

GET 通常无请求体，POST/PUT/DELETE 常用

## 二、HTTP 响应结构
HTTP 响应由 状态行、响应头、空行、响应体 四部分组成：
### 状态行
包含：协议版本、状态码、状态描述信息
### 响应头
以 Key: Value 形式返回服务端信息、缓存、类型等
### 空行
固定分隔符，区分响应头与响应体
### 响应体
服务端返回给客户端的数据，如 HTML、JSON、图片、文件等

# HTTP 状态码分类（精简整理版）
HTTP 状态码为**三位数字**，按首位分为 5 类：

## 1xx：提示信息
- 表示请求已接收，需要继续处理
- 典型：**101** 切换协议（常用于 WebSocket）

## 2xx：成功
- 服务端已正常接收并处理请求
- **200 OK**：请求成功（最常用）
- **204 No Content**：成功，但无响应体
- **206 Partial Content**：部分内容（分块下载、断点续传）

## 3xx：重定向
- 需客户端进一步操作才能完成请求
- **301**：永久重定向（旧地址废弃，推荐用 HTTPS 场景）
- **302**：临时重定向（资源仍存在，临时跳转）
- **304**：缓存重定向（资源未修改，使用本地缓存）

## 4xx：客户端错误
- 请求报文、权限、地址等出错
- **400 Bad Request**：请求报文错误（通用）
- **403 Forbidden**：无权限，拒绝访问
- **404 Not Found**：资源不存在

## 5xx：服务端错误
- 服务端处理请求时发生异常
- **500 Internal Server Error**：服务器内部错误（通用）
- **503 Service Unavailable**：服务暂时不可用（过载/维护）

# http和https的区别？

1. **传输安全性**
- HTTP 是**明文传输**，数据易被窃听、篡改；
- HTTPS 在 HTTP 与 TCP 之间加入 **SSL/TLS 加密层**，实现加密传输，更安全。

2. **连接过程**
- HTTP 完成 **TCP 三次握手**后即可直接传输数据；
- HTTPS 在 TCP 握手后，还需进行 **SSL/TLS 握手**，协商加密方案，再传输数据。

3. **默认端口**
- HTTP 默认端口：**80**
- HTTPS 默认端口：**443**

4. **身份认证**
- HTTP 无身份验证；
- HTTPS 需要向 **CA 机构申请证书**，用于验证服务器身份，防止伪装钓鱼。

# GET 与 POST 请求区别
1. **用途不同**
   - GET 用于**从服务器获取资源**，是查询操作；
   - POST 用于**向服务器提交数据**，常用于新增、修改等操作。

2. **安全性与幂等性**
   - GET 是只读操作，相对安全且**幂等**（多次执行结果一致）；
   - POST 会修改服务器数据，不安全且**不幂等**。

3. **参数位置与大小限制**
   - GET 参数拼接在 **URL** 中，受浏览器长度限制，数据量较小；
   - POST 参数放在**请求体**中，理论上无大小限制，可传输更多数据。

# HTTP 请求处理完整流程
1. **解析请求行**
   读取请求第一行，按空格拆分得到：**请求方法**、**请求URI**、**HTTP协议版本**。

2. **解析请求头**
   逐行读取 `Key: Value` 格式的请求头，直到遇到**空行**结束。

3. **解析请求体**
   根据 `Content-Length` 读取对应长度的数据，并依据 `Content-Type`（如 application/json）进行格式解析。

4. **请求路由与业务处理**
   根据请求URI匹配后端接口，执行对应业务逻辑。

5. **构造响应返回**
   按照固定格式组装响应数据：**状态行 + 响应头 + 空行 + 响应体**，返回给客户端。

# 什么是TCP协议？
TCP是传输层协议，给应用层提供可靠的服务，保证数据从一台机器到另一台机器不会丢，不乱序，不重复。

# TCP 与 UDP 区别
1. **连接方式**
   - TCP：面向连接，需通过**三次握手**建立可靠连接后再传输。
   - UDP：无连接，无需握手，直接发送数据报。

2. **传输方式**
   - TCP：以**字节流**形式传输，保证顺序、无丢失、无重复。
   - UDP：以**数据报包**为单位独立发送，不保证顺序与完整性。

3. **可靠性机制**
   - TCP：具备**超时重传、流量控制、拥塞控制**，保证数据可靠传输。
   - UDP：无可靠性保证，可能丢包、乱序。

4. **性能与适用场景**
   - TCP：可靠但开销大，适合文件传输、网页、接口等对准确性要求高的场景。
   - UDP：包头更小、无连接开销，**速度更快、实时性更强**，适合直播、视频通话、游戏等场景。

# TCP 粘包拆包
## 1. 现象
- **粘包**：发送多条消息（如 AB、CD），接收方一次性读到合并数据（ABCD），无法区分消息边界。
- **拆包**：发送一条完整消息（如 ABCDEF），接收方分成多次收到（ABC、DEF），一条消息被拆开。

## 2. 原因
- TCP 是**面向字节流**的协议，没有天然消息边界。
- Nagle 算法会合并小数据包，提高传输效率，导致粘包。
- 数据超出**发送缓冲区**或**MTU（最大传输单元）**，会自动拆包发送。

## 3. 解决方案
- **固定长度**：每条消息长度一致，按固定长度截取。
- **分隔符**：在消息末尾加特殊符号（如 \n、\r\n）作为边界。
- **消息头+消息体**：消息头中存储消息体长度，按长度读取。

# TCP 三次握手
## 一、三次握手过程
TCP 三次握手的核心目的：**确认双方收发能力正常，并同步初始化序列号（ISN）**，保证连接可靠建立。

1. **第一次握手（客户端 → 服务端）**
   - 客户端发送 `SYN=1` 报文，随机初始序列号 `seq = x`
   - 客户端状态：`CLOSED → SYN_SENT`
   - 此报文**不能携带数据**

2. **第二次握手（服务端 → 客户端）**
   - 服务端回复 `SYN=1, ACK=1`
   - 确认号 `ack = x + 1`，同时携带自己的随机序列号 `seq = y`
   - 服务端状态：`LISTEN → SYN_RCVD`

3. **第三次握手（客户端 → 服务端）**
   - 客户端发送 `ACK=1`，确认号 `ack = y + 1`
   - 客户端状态：`SYN_SENT → ESTABLISHED`
   - 服务端收到后：`SYN_RCVD → ESTABLISHED`
   - 本次报文**可以携带数据**

---

## 二、为什么必须是三次握手？

### 1. 防止历史重复连接导致资源浪费（首要原因）
网络中可能存在**延迟、滞留的旧 SYN 包**。
如果只有两次握手：
- 服务端收到旧 SYN 就直接建立连接
- 客户端发现是过期请求会拒绝，但服务端已占用连接资源
  三次握手可以让客户端有机会回复 `RST` 复位报文，**阻止无效连接建立**。

### 2. 必须同步双方的初始化序列号（ISN）
TCP 可靠传输依赖序列号：
- 客户端 `seq=x` → 服务端 `ack=x+1` 确认
- 服务端 `seq=y` → 客户端 `ack=y+1` 确认

要完成**双向序列号同步**，至少需要三次交互：
客户端发 → 服务端回应+发 → 客户端回应
少一次都无法保证双方收发能力与序列号都确认完毕。

# TCP 四次挥手

## 一、基本说明
- 角色：**客户端**（主动关闭方）、**服务端**（被动关闭方）
- 初始状态：双方均为 `ESTABLISHED`

---

## 二、详细过程
1. **第一次挥手**
   客户端 → 服务端：发送 FIN 包（seq=x）
   - 客户端状态：`ESTABLISHED → FIN-WAIT-1`

2. **第二次挥手**
   服务端 → 客户端：回复 ACK 包（ack=x+1）
   - 客户端状态：`FIN-WAIT-1 → FIN-WAIT-2`
   - 服务端状态：`ESTABLISHED → CLOSE-WAIT`

3. **第三次挥手**
   服务端 → 客户端：发送 FIN 包（seq=y）
   - 服务端状态：`CLOSE-WAIT → LAST-ACK`

4. **第四次挥手**
   客户端 → 服务端：回复 ACK 包（ack=y+1）
   - 客户端状态：`FIN-WAIT-2 → TIME-WAIT`
     等待 **2MSL** 后，最终变为 `CLOSED`
   - 服务端状态：`LAST-ACK → CLOSED`

---

## 三、记忆口诀
客发 FIN → FIN1
服回 ACK → 客 FIN2，服 CLOSE
服发 FIN → 服 LAST
客回 ACK → 客 TIME，服 CLOSED

---

## 四、高频考点
- **TIME-WAIT**：主动关闭方最后等待 2MSL，防止服务端未收到 ACK 而重传 FIN。
- **CLOSE-WAIT**：表示服务端已收到关闭请求，等待应用层处理完毕后再发送 FIN。

# TCP/IP 四层模型
1. **应用层**
   为应用程序提供网络服务，如 HTTP、HTTPS、FTP、DNS、WebSocket 等。

2. **传输层**
   负责端到端的数据传输控制，主要协议为 **TCP**、**UDP**。

3. **网络层**
   负责主机之间的寻址与路由选择，核心协议是 **IP**，还有 ARP、ICMP 等。

4. **网络接口层**
   负责数据在物理网络中的传输，包含设备驱动、网卡、物理线路等。

# OSI 七层模型
1. **应用层**
   为应用提供网络服务，如 HTTP、HTTPS、DNS、FTP 等。

2. **表示层**
   负责数据格式转换、加密、解密、压缩、解压缩。

3. **会话层**
   建立、管理、终止应用之间的会话连接。

4. **传输层**
   负责端到端数据传输与控制，主要协议 TCP、UDP。

5. **网络层**
   负责主机间寻址与路由，核心协议 IP、ARP、ICMP。

6. **数据链路层**
   将网络层数据封装成帧，控制物理层传输，差错检测。

7. **物理层**
   传输原始比特流，定义电气、物理接口规范（网线、网卡等）。

# Cookie、Session、Token 区别
1. **Cookie**
- 由**服务端生成**，保存在**浏览器**
- 浏览器每次请求会自动携带 Cookie
- 服务端通过 Cookie 识别用户、读取数据

2. **Session**
- 是**服务端存储用户状态**的机制
- 每个 Session 对应一个唯一 **SessionID**，存在浏览器 Cookie 中
- 客户端请求时带上 SessionID，服务端据此找到对应用户数据

3. **Token**
- 本质是一段**加密字符串**，用于身份认证与权限校验
- 用户信息经加密后存在 Token 里
- 服务端收到 Token 解密后，即可直接获取用户信息，无需再查库

# JWT 详解
JWT 是一种**无状态认证令牌**，以加密字符串形式安全传输 JSON 格式的用户信息。

## 一、JWT 结构
由三部分通过 `.` 拼接而成：
**Header（头部）.Payload（载荷）.Signature（签名）**

1. **Header（头部）**
- 声明令牌类型为 JWT、签名加密算法（如 HS256）
- 经 Base64URL 编码得到第一部分

2. **Payload（载荷）**
- 存储用户信息、过期时间等业务数据
- 同样经 Base64URL 编码得到第二部分

3. **Signature（签名）**
- 将前两部分编码结果 + 服务端密钥，使用指定算法加密生成
- 用于校验数据在传输过程中是否被篡改

## 二、校验原理
服务端收到 JWT 后：
1. 拆分出 Header、Payload、Signature
2. 用相同算法和密钥重新计算签名
3. 与原 Signature 对比，一致则未被篡改，否则无效

## 三、生成流程
1. 构造 Header、Payload
2. 分别做 Base64URL 编码
3. 拼接两部分，用密钥 + 算法生成签名
4. 最终拼接为：`Header.Payload.Signature`

# DNS 域名解析流程
DNS 的作用是**将域名解析为 IP 地址**，完整解析流程如下：

1. **浏览器缓存查询**
   先查找浏览器自身缓存中是否有该域名对应的 IP。

2. **本地 Host 文件查询**
   浏览器缓存无结果，则读取操作系统的 `hosts` 文件。

3. **本地 DNS 服务器查询**
   Host 无记录时，向运营商本地 DNS 服务器发起查询。

4. **根 DNS 服务器查询**
   本地 DNS 无缓存，询问根 DNS，根 DNS 告知对应**顶级域名 DNS** 地址。

5. **顶级域名 DNS 查询**
   本地 DNS 访问顶级域名 DNS，获取对应**权威 DNS** 地址。

6. **权威 DNS 查询**
   本地 DNS 向权威 DNS 查询，最终拿到目标域名的 IP。

7. **返回并缓存**
   本地 DNS 将 IP 返回给浏览器，同时**浏览器与本地 DNS 均缓存该记录**。

# CDN 简介
CDN 是由**多地域分散部署的服务器**构成的分布式网络。

- 工作原理：用户请求资源时，会被**就近调度到距离最近的节点服务器**获取缓存内容。
- 优势：
   1. 就近访问，**降低网络延迟**，提升访问速度
   2. 分担源站流量，**减轻源站压力**
   3. 源站故障时，缓存节点可继续提供服务，**提高系统可用性**

# 浏览器输入网址到页面展示全过程

当我们在浏览器输入网址并按下回车后，整体流程如下：

### 1.  **URL 解析**
   浏览器首先会对输入的 URL 进行解析，拆分出其中的协议、域名、资源路径、端口号等关键信息，明确要访问的目标资源。

### 2. **DNS 域名解析**
   浏览器需要将域名解析成对应的 IP 地址才能发起请求。
   查找顺序依次是：先查浏览器自身缓存 → 没有的话查操作系统缓存 → 再没有就查看本地 hosts 文件 → 仍然没有就向本地 DNS 服务器发起查询。
   如果本地 DNS 服务器也没有缓存记录，它会依次向根域名服务器、顶级域名服务器、权威域名服务器进行迭代查询，最终拿到目标 IP 地址，并返回给浏览器，同时浏览器和本地 DNS 都会缓存这条记录。

### 3. **建立可靠连接**
   拿到 IP 地址和端口号后，浏览器会与服务器进行 **TCP 三次握手**，建立可靠的传输连接。
   如果是 HTTPS 协议，在 TCP 三次握手之后，还会进行 **TLS 握手**，协商加密套件、交换密钥，确保后续数据传输的安全性。

### 4. **发送 HTTP 请求并接收响应**
   连接建立完成后，浏览器构造并发送 HTTP 请求报文，服务端接收到请求后进行业务处理，然后返回 HTTP 响应报文给浏览器。

### 5. **浏览器解析渲染页面**
   浏览器接收到响应数据后，开始解析 HTML 构建 DOM 树，解析 CSS 构建 CSSOM 树，结合生成渲染树，再经过布局、绘制等步骤，最终将页面呈现到屏幕上。

### 6. **断开 TCP 连接**
   当数据传输完成、页面加载结束后，客户端和服务端会通过 **TCP 四次挥手** 的过程，正常断开连接，释放相关资源。

# 四种常见认证方式
### 1. **Session-Cookie 认证**
   基于服务端会话的认证方式。用户登录成功后，服务端创建一个 Session 存储用户状态与信息，并生成唯一的 Session ID。这个 Session ID 会通过 Cookie 写入浏览器。后续前端每次请求时，浏览器都会自动在 Cookie 中带上这个 Session ID，服务端根据 ID 找到对应的 Session，从而识别用户身份、获取登录状态。

### 2. **Token 认证**
   无状态的认证方式。用户登录成功后，服务端将用户信息、权限、过期时间等数据加密生成一串令牌（Token，如 JWT），返回给前端。前端将 Token 保存在浏览器本地（localStorage、sessionStorage 或 Cookie），之后每次请求都在请求头中主动携带 Token。服务端收到后直接解密验证，不需要在服务端存储会话状态，扩展性更强，适合分布式、前后端分离项目。

### 3. **OAuth 2.0 认证**
   一种第三方开放授权协议。用户不需要在当前系统输入账号密码，而是通过跳转到微信、QQ、GitHub 等第三方平台进行登录授权。第三方平台验证通过后，会返回一个授权码或令牌给当前系统，当前系统凭此令牌向第三方平台获取用户信息，完成登录。核心是**授权而不是直接账号密码登录**，安全且便捷。

### 4. **API Key 认证**
   适用于接口调用、开放平台、服务间调用的简单认证方式。由后端预先为调用方生成一个唯一的、长期有效的密钥（API Key / AppKey），有的还会搭配密钥密钥（Secret）。调用方在请求时把 API Key 放在请求头、参数或签名中，服务端收到后与数据库或配置中的密钥比对，比对一致则鉴权通过。优点是简单直接、易于使用；缺点是安全性依赖密钥保护，一般配合签名、IP 白名单、权限控制一起使用。
